# Aura Harness Remediation Plan

This document turns the `aura-harness` review findings into concrete work items.

Scope:
- `crates/aura-harness`
- Shared semantic lane only
- Frontend-conformance-only mechanics may remain quarantined, but they must not leak into shared semantic execution

Cleanup policy:
- Delete obsolete code instead of wrapping it
- Prefer direct replacement over compatibility shims
- Do not preserve backwards-compatibility branches inside `aura-harness` for these fixes
- Do not add migration code, compatibility adapters, or legacy fallback paths
- If an old path is no longer the sanctioned architecture, remove it rather than deprecating it in place

Out of scope:
- Rewriting unrelated network-lab tooling unless needed to remove one of the issues below
- Product-side semantic ownership changes outside the minimum bridge/contract changes needed by the harness

## 1. Remove Harness-Owned Semantic State Duplication

Problem:
- The harness keeps its own semantic state machine and convergence ledger in executor-local state.
- This duplicates product truth and lets the harness reject or permit flows based on its own reconstruction instead of authoritative upstream state.

Tasks:
- [x] Inventory every use of `shared_flow_state` and `pending_convergence` in `crates/aura-harness/src/executor.rs`
- [x] Classify each use as either:
  - [x] authoritative observation that should come from typed `UiSnapshot` / runtime facts
  - [x] harness-local orchestration bookkeeping that is still legitimate
  - [x] duplicate semantic ownership that must be removed
- [x] Delete harness-local semantic phase gating for shared semantic actions where product state already exposes typed readiness / operation / runtime-fact contracts
- [x] Delete the old harness-owned semantic lifecycle bookkeeping once replacement typed checks exist
- [x] Replace local phase prerequisites with typed snapshot-based precondition checks
- [x] Replace local convergence bookkeeping with typed post-operation barriers derived directly from shared action contracts plus authoritative observations
- [x] Keep only executor-local orchestration metadata that cannot be confused with product semantic truth
- [x] Remove dead enums, transitions, helper functions, and fields after the duplicate semantic state machine is gone

Success criteria:
- [x] Shared semantic execution no longer depends on `SharedFlowState` as a source of truth for legality of semantic actions
- [x] The harness does not maintain a parallel semantic lifecycle graph for account/contact/channel/message progress
- [x] Any remaining executor-local state is clearly orchestration-only and cannot be mistaken for authoritative semantic truth
- [x] A reviewer can trace every shared semantic precondition back to typed product state, not harness-reconstructed state

## 2. Eliminate Heuristic Channel Identity Re-Materialization

Problem:
- The harness reconstructs authoritative channel identity from names, cached strings, and runtime-event heuristics like “unique shared channel with member_count > 1”.
- That is a weak-to-strong upgrade and breaks ownership discipline.

Tasks:
- [x] Inventory every path that writes or reads:
  - [x] `current_channel_name`
  - [x] `current_channel_id`
  - [x] `channel_name_by_id`
  - [x] `resolve_channel_id_for_shared_name`
  - [x] `unique_authoritative_shared_channel_id`
  - [x] `capture_authoritative_channel_id`
  - [x] `capture_unique_shared_channel_id`
- [x] Define the sanctioned typed channel-binding handoff for shared semantic flows
- [x] Require shared semantic flows to carry typed channel/context bindings from the authoritative submission/result path instead of re-deriving them later
- [x] Remove fallback logic that guesses a channel id from ambient runtime events
- [x] Remove “current channel” string caches where a typed binding or typed selected-channel reference should be carried instead
- [x] Fail closed when the harness does not possess an authoritative channel binding
- [x] Delete obsolete name-to-id repair helpers and any now-unused cached string maps

Success criteria:
- [x] No shared semantic flow upgrades a channel name or ambient runtime event into an authoritative channel id by heuristic
- [x] `unique_authoritative_shared_channel_id`-style logic is removed from shared semantic execution
- [x] Channel targeting after create/join/invite uses a typed handoff object or typed authoritative reference
- [x] Missing channel ownership/binding produces a direct typed failure instead of a repair attempt

## 3. Make Operation Handle Transfer Mandatory

Problem:
- Handle transfer is optional in the local PTY command plane.
- The harness repairs missing handles by polling `ui_snapshot()` and inferring which operation submission “must be” the right one.

Tasks:
- [x] Inventory every `send_harness_command(...) -> Option<...>` call site in `crates/aura-harness/src/backend/local_pty.rs`
- [x] Inventory every use of `wait_for_operation_submission(...)`
- [x] Change the shared semantic command plane so parity-critical submissions always return a typed operation handle when the command contract requires one
- [x] Tighten local PTY receipt typing so `Accepted` cannot omit handles for parity-critical operations
- [x] Remove fallback polling-based handle inference for parity-critical shared semantic submissions
- [x] Keep any handle-optional path only where the semantic contract explicitly permits no handle
- [x] Add compile-time or API-shape pressure so future parity-critical submissions cannot silently regress to optional handles
- [x] Delete `Option`-based handle plumbing that only exists for backwards compatibility
- [x] Delete any legacy receipt variants or branches that preserve missing-handle acceptance for parity-critical flows

Success criteria:
- [x] Shared semantic submissions do not rely on `wait_for_operation_submission(...)` as a repair path
- [x] PTY/browser shared semantic submissions either return the required handle or fail
- [x] Operation ownership transfer is explicit at submission time
- [x] There is no time-window inference of “the likely matching operation instance” for parity-critical flows

## 4. Remove Raw Screen / Text Heuristics from Shared Semantic Waits

Problem:
- Shared semantic waits still succeed through raw screen containment and renderer text fallback after typed observation did not prove the condition.
- This keeps the primary lane renderer-driven.

Tasks:
- [x] Inventory all shared semantic wait paths in `executor.rs`, `coordinator.rs`, and backend adapters
- [x] Separate:
  - [x] typed semantic observation
  - [x] diagnostic screen capture
  - [x] frontend-conformance-only raw mechanics
- [x] Remove semantic success paths that depend on:
  - [x] `message_contains_authoritative_screen(...)`
  - [x] diagnostic screen text fallback after typed wait failure
  - [x] raw `wait_for` screen pattern polling in shared semantic execution
- [x] Keep screen/DOM capture for diagnostics only, never as a success condition in the shared semantic lane
- [x] Ensure shared semantic waits are satisfied only by:
  - [x] typed `UiSnapshot`
  - [x] typed operation handle state
  - [x] typed readiness / quiescence
  - [x] typed runtime facts / shared action barriers
- [x] Make any remaining raw screen or DOM waits explicit frontend-conformance-only APIs
- [x] Delete shared-semantic helper paths whose only purpose was to preserve old raw-screen success behavior

Success criteria:
- [x] Shared semantic waits cannot return success from raw screen or DOM text matching
- [x] Screen/DOM data is used only for debugging, failure attribution, or conformance-only paths
- [x] The shared semantic lane remains valid even if renderer text changes while typed semantic state remains stable
- [x] A reviewer can grep shared semantic execution paths and not find raw `wait_for`-style success conditions

## 5. Replace JSON-and-Stringly Boundaries with Typed Surfaces

Problem:
- Core harness boundaries still use `serde_json::Value`, raw string fields, and ad hoc payload decoding.
- This erodes the type guarantees the harness is supposed to preserve.

Tasks:
- [x] Inventory all shared semantic lane uses of:
  - [x] `serde_json::Value`
  - [x] untyped `payload.get(...)`
  - [x] stringly event fields
  - [x] screen field fallback chains
- [x] Split typed semantic tool responses from generic/raw diagnostic responses
- [x] Introduce typed result structs for shared semantic tool operations that currently return `ToolResponse::Ok { payload: Value }`
- [x] Introduce typed event-detail structures for harness events used by shared semantic orchestration
- [x] Replace generic payload parsing in shared semantic executor code with typed decode functions
- [x] Keep raw JSON only for:
  - [x] CLI/debug output
  - [x] quarantined compatibility paths
  - [x] explicitly generic tooling where no semantic contract is claimed
- [x] Remove screen-field fallback chains from typed paths
- [x] Delete generic JSON wrappers in shared semantic code once typed equivalents exist
- [x] Delete legacy field-peeking helpers that remain only for compatibility with older payload shapes

Success criteria:
- [x] Shared semantic executor/orchestrator paths do not depend on ad hoc `payload.get("...")` extraction
- [x] Shared semantic event records have typed detail structures, or an equivalent typed boundary, for orchestrator-critical data
- [x] Tool API surfaces consumed by shared semantic execution are typed end to end
- [x] Raw JSON remains only at diagnostics / transport edges, not at semantic decision points

## 6. Remove Fixed Time Windows Where Ownership Transfer Exists

Problem:
- The harness uses fixed sleeps, settle windows, polling intervals, and short recovery loops in places where a proper typed event or ownership transfer should exist.
- Some of these are legitimate infrastructure timeouts; others are architectural escape hatches.

Tasks:
- [x] Inventory every fixed wait in shared semantic paths, especially:
  - [x] PTY snapshot settling and header recovery
  - [x] readiness polling
  - [x] list selection convergence loops
  - [x] operation submission polling
  - [x] coordinator `wait_for` loops
  - [x] browser readiness / RPC timeout margin logic
- [x] Classify each timing-based mechanism as one of:
  - [x] infrastructural transport timeout that is acceptable
  - [x] diagnostic-only delay that must not affect semantic success
  - [x] ownership-transfer gap that must be replaced by a typed event/receipt/handle
  - [x] renderer-conformance-only heuristic that must stay quarantined
- [x] Replace ownership-gap timers with explicit typed publication or receipt contracts
- [x] Move any unavoidable wall-clock logic behind sharply named infrastructure boundaries
- [x] Document every remaining time-based loop with owner, reason, and why a typed contract does not yet exist
- [x] Delete stale settle/recovery windows that no longer serve a necessary infrastructure purpose

Success criteria:
- [x] No shared semantic ownership transfer depends on a settle loop or heuristic polling window
- [x] Remaining timers in shared semantic code are infrastructure-only and not semantic truth sources
- [x] Renderer stabilization waits are quarantined from the shared semantic lane
- [x] The diff removes or sharply reduces fixed sleeps in shared semantic execution paths

## 7. Collapse Duplicate Observation Planes

Problem:
- The harness exposes and consumes multiple overlapping observation systems:
  - typed `UiSnapshot`
  - raw screen text
  - normalized screen text
  - DOM snapshot text
  - runtime-event scanning
- These overlap in ways that let weaker planes repair stronger ones.

Tasks:
- [x] Define a single observation hierarchy for the harness:
  - [x] authoritative semantic projection
  - [x] typed event stream
  - [x] diagnostics only
- [x] Update backend traits so shared semantic execution consumes the authoritative semantic plane first-class
- [x] Ensure raw screen / DOM surfaces cannot masquerade as authoritative observation in trait signatures
- [x] Remove fallback ordering like `authoritative_screen -> normalized_screen -> screen` from semantic code
- [x] Make diagnostic screen capture explicit in function names and return types
- [x] Delete duplicate observation helpers that keep the old mixed-plane behavior alive

Success criteria:
- [x] Shared semantic execution has one primary observation plane
- [x] Raw screen and DOM planes cannot silently substitute for typed semantic observation
- [x] Backend trait signatures make the distinction obvious
- [x] Failure diagnostics still include raw captures, but success conditions do not depend on them
